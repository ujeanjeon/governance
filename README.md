# SKT CX Governance Hub

SKT Next Channel UX 거버넌스 문서 사이트.  
배포 주소: **https://nova-governance.vercel.app**

---

## 사이트 구조

```
governance/
├── index.html              ← 홈 (문서 카드 목록)
├── UXP/
│   └── UXP_governance.html ← UX 원칙 (여정 7단계 × 16패턴)
├── UXPT/
│   └── UXPT_governance.html ← UX 패턴 (버튼·로딩·네비 등 56패턴)
├── VOT/
│   └── VOT_governance.html ← Voice & Tone (DEF·RUL·SIT·SEG)
├── AIG/
│   └── AIG_governance.html ← AI 가이드라인 (T Agent·Use Case·비주얼)
├── CV/
│   └── CV_governance.html  ← 서비스 핵심 가치 (CV01~CV04)
├── TK/                     ← Confluence 원본 참고자료 (배포 안 됨, 작업 맥락용)
└── llms.txt                ← AI 참조용 인덱스
```

각 `*_governance.html`이 실제 배포되는 문서 본체입니다.  
`TK/` 폴더는 원본 기획 자료(Confluence 저장본)로, 각 거버넌스 패턴의 근거 자료입니다.

## 수정 및 배포 방법

### 저장소 처음 받기
```bash
git clone https://github.com/ujeanjeon/governance.git
cd governance
```

### 파일 수정 후 배포
```bash
git add .
git commit -m "VOT 컴포넌트 규칙 추가"
git push
```

push 후 1~2분 내에 **https://nova-governance.vercel.app** 자동 반영됩니다.  
Vercel 계정·CLI 설치 불필요.

## VOT 인수인계 — 클라이언트 피드백 (2026.06.23)

> 8월부터 SB Studio를 통한 스토리보드 산출 시작 예정.  
> 현재 VOT 수준으로는 스토리보드 내 라이팅 품질을 보장하기 어렵다는 피드백.

---

### 배경: 왜 지금 VOT로는 부족한가

현재 VOT는 **원칙 중심**으로 작성되어 있다.  
"능동태를 써라", "결론을 먼저 말해라", "50자 이내로 써라" 같은 규칙이 잘 정리되어 있지만,  
이것만으로는 AI(SB Studio)가 실제 컴포넌트에 들어갈 카피를 생성할 때 활용하기 어렵다.

AI가 카피를 잘 생성하려면 원칙 외에 세 가지가 더 있어야 한다.

1. **컴포넌트별 라이팅 규칙** — "이 컴포넌트엔 이 구조로 써야 한다"는 틀
2. **교정 기준 데이터** — "이 표현은 틀렸고, 이렇게 고쳐야 한다"는 예시 셋
3. **외부 가이드라인 통합** — 고객언어연구팀이 보유한 전사 기준과 일치시키기

### 추가 필요한 작업 (우선순위 순)

#### 1순위: VOT_CMP — 컴포넌트별 라이팅 규칙 신설

**왜 필요한가**  
현재 `VOT_RUL_1`(텍스트 길이 규칙)은 컴포넌트별 글자 수 상한선만 있다.  
SB Studio가 스토리보드를 생성할 때 참조하는 건 이 규칙인데, 길이만 알아서는 카피 구조를 모른다.  
예를 들어 "AIA Advice 메인 메시지는 50자 이내"라는 기준만 있고, 그 50자를 어떤 구조로 채워야 하는지가 없다.

**무엇을 추가해야 하는가**  
각 컴포넌트마다 아래 세 가지를 정의해야 한다.

- **구조 공식** — 카피를 어떤 순서·형태로 구성할지. 예: AIA Advice = `[상태 인식] + [행동 제안]` 2문장 구조
- **DO / DON'T** — 실제 예시로 된 맞는 표현과 틀린 표현
- **Copy Template** — AI가 변수만 채우면 완성되는 문장 틀. 예: `"{이름}님, {상태}예요. {행동}하면 {결과}예요."`

**대상 컴포넌트 7종**

| 컴포넌트 | 구조 공식 (작성 방향) |
|---|---|
| **Micro Task 카드 레이블** | `[동사]+[목적어]` 단문. 조건절·수식어 금지. 예: "데이터 선물하기" O / "데이터를 선물하실 수 있어요" X |
| **Micro Task 카드 컨텍스트** | 레이블을 뒷받침하는 안심 정보 1줄. 숫자 또는 상태 중심. 예: "잔여 1.2GB · 만료 D-3" |
| **AIA Advice** | `[상태 인식 1문장] + [행동 제안 1문장]`. 결론(행동 제안)을 2번째 문장에. 개인화 변수({이름}, {수치})는 1번째 문장에 삽입 |
| **Offer 카드 headline** | 절감 금액 숫자 중심. `"월 N원 절약"` 패턴 고정. 수식어 금지 |
| **Offer 카드 sub** | headline의 근거를 1줄로. `"{이름}님 {이용 패턴} 기준"` 형식. 고객명 포함 시 신뢰도 상승 |
| **토스트 / 스낵바** | `[완료 동사]+[결과 상태]` 구조. 수동형 금지. 예: "변경했어요. 다음 달 1일부터 적용돼요" |
| **오류 메시지** | `[원인 1문장] + [다음 행동 1문장]` 2줄 구조. 원인은 시스템 탓이 아닌 상황 설명으로. 예: "연결이 잠깐 끊겼어요. 다시 시도해 주세요" |

참고 레퍼런스: [신한카드 UX 라이팅 가이드](https://ux.shinhan.com/155fe4737/p/779d89---)  
(컴포넌트별 구조·금지표현·예시를 체계적으로 다룬 레퍼런스)

#### 2순위: VOT_TRM — 용어집 & 교정셋 신설

**왜 필요한가**  
원칙만 있으면 AI가 그럴듯하지만 틀린 카피를 생성한다.  
"지침만으로 테스트했을 때 라이팅 퀄리티가 잘 나오지 않았다"는 게 클라이언트가 직접 경험한 결과다.  
교정 기준 데이터(as-is → to-be 쌍)가 있어야 AI가 실수를 반복하지 않는다.

**무엇을 만들어야 하는가**

| 항목 | 설명 | 예시 |
|---|---|---|
| **권장어 매핑 테이블** | 금지 표현과 그 대체 표현을 1:1로 정리 | "완료되었습니다" → "완료됐어요" |
| **as-is / to-be 교정셋** | 실제 기존 카피와 개선된 카피를 쌍으로 30~50개 | "요금제 변경이 완료되었습니다" → "요금제를 변경했어요. 다음 달 1일부터 적용돼요" |
| **정규표현식 패턴** | 통화·날짜·시간의 올바른 표기 형식 | 금액: `[0-9,]+원` / 날짜: `[0-9]+월 [0-9]+일` / 기간: `D-[0-9]+` |
| **용어집** | 서비스 내 고유 용어의 표기 기준 | "T world" O / "T World", "티월드" X |

**수신 일정 및 담당**

| 자료 | 출처 | 수신 시기 |
|---|---|---|
| '사람잡는 글쓰기' | 고객언어연구팀 (전사 UX 라이팅 기준) | 메일 수신 예정 (곧) |
| 디지털 UX언어 가이드 | 고객언어연구팀 (제작 중) | **2026년 8월** |
| 교정셋·용어집·매핑 테이블 | 내부 수집 중 | 추후 전달 예정 |

> **이후 활용 방향**: 이 데이터는 RAG화해서 SB Studio API에 연동 예정 (기획2팀 현규님 협업).  
> 스토리보드 카피 생성 시 VOT 거버넌스 원칙 검토 → RAG 교정셋 검토를 순서대로 통과하는 2-pass 구조 목표.  
> 데이터 수신 시 `VOT_TRM` 섹션 신설 후 `llms.txt`도 함께 업데이트할 것.

#### 3순위: llms.txt 업데이트

VOT_CMP, VOT_TRM 섹션 추가 후 `llms.txt` 내 VOT 항목에 패턴 ID 목록 추가.  
llms.txt는 AI(SB Studio 포함)가 이 사이트를 참조할 때 읽는 인덱스 파일이므로 반드시 동기화할 것.
